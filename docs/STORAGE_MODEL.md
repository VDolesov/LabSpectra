# LabSpectra — Модель хранения данных

Документ описывает, **как и где** LabSpectra хранит данные. Хранилище —
обычные папки и файлы на локальном диске, доступные через Проводник.
Управляет ими только приложение.

## 1. Структура папки `lab_data`

```
lab_data/
├── registry.xlsx            # человекочитаемый реестр всех анализов
├── samples/                 # папки отдельных анализов
│   ├── IR-2026-001/
│   ├── IR-2026-002/
│   └── ...
├── backups/                 # резервные копии (архивы lab_data / registry)
├── logs/                    # журнал действий приложения (app.log)
├── .trash/                  # удалённые/замещённые файлы (мягкое удаление)
└── .lock                    # служебный lock-файл на время записи
```

- `lab_data/` — **корень всех данных**. Путь к нему хранится в настройках
  приложения; по умолчанию рядом с исполняемым файлом или в выбранной
  пользователем папке.
- `samples/` — единственное место, где лежат данные анализов.
- `backups/`, `logs/`, `.trash/`, `.lock` — служебные, приложение создаёт их само.
- Всё, что нужно для восстановления, находится **внутри `lab_data/`** —
  достаточно скопировать эту папку, чтобы перенести/сохранить всю базу.

## 2. Структура папки одного анализа

```
samples/IR-2026-001/
├── card.json                # машиночитаемая карточка анализа (источник истины)
├── photos/                  # фотографии образца
│   ├── photo_1.jpg
│   └── photo_2.jpg
├── spectra/                 # ИК-спектры
│   ├── spectrum_1.pdf
│   └── spectrum_1.txt
└── reports/                 # отчёты по анализу
    └── report.pdf
```

Правила:

- Имя папки анализа **совпадает с ID анализа** (`IR-2026-001`).
- Внутри всегда три подпапки: `photos/`, `spectra/`, `reports/` — создаются
  при создании анализа, даже если пока пустые.
- `card.json` лежит в корне папки анализа.
- Никаких файлов данных вне этих трёх подпапок приложение не размещает.

## 3. Назначение `registry.xlsx`

`registry.xlsx` — **основной человекочитаемый реестр**. Каждая строка = один
анализ/образец. Его можно открыть в Excel/LibreOffice без приложения и
прочитать глазами или распечатать.

Колонки (фиксированный порядок):

| № | Колонка                  | Описание                                  |
|---|--------------------------|-------------------------------------------|
| 1 | ID анализа               | `IR-2026-001`, ключ строки                |
| 2 | дата анализа             | дата проведения анализа                   |
| 3 | продукт                  | наименование продукта                     |
| 4 | партия                   | номер партии                              |
| 5 | название образца         | человекочитаемое имя образца              |
| 6 | описание                 | описание образца/задачи                   |
| 7 | краткий результат анализа| короткий вывод                            |
| 8 | статус                   | напр. `новый` / `в работе` / `готов`      |
| 9 | путь к папке анализа     | относительный путь к папке анализа        |
| 10| ссылки на фотографии     | относительные пути к фото (через `;`)     |
| 11| ссылки на спектры        | относительные пути к спектрам (через `;`) |
| 12| ссылки на отчёты         | относительные пути к отчётам (через `;`)  |
| 13| комментарий              | произвольный комментарий                  |
| 14| дата создания            | когда запись создана приложением          |
| 15| дата изменения           | когда запись последний раз менялась       |

Свойства:

- Реестр — **производный индекс**: всегда восстановим из совокупности
  `card.json` (см. ARCHITECTURE.md, разделы 4 и 7).
- Порядок и имена колонок фиксированы; версия схемы хранится отдельно
  (служебный лист/ячейка), чтобы приложение умело читать старые файлы.
- Заполняется и обновляется **только приложением**. Ручное редактирование
  допустимо как просмотр/правка отдельных полей, но не как основной способ
  ведения учёта (см. MVP, риски).

## 4. Назначение `card.json`

`card.json` — **машиночитаемая карточка анализа** внутри его папки. Она:

- является **источником истины** для данных конкретного анализа;
- используется для **восстановления** `registry.xlsx` при повреждении;
- хранит полную информацию, включая служебные поля, которых может не быть в Excel.

Пример:

```json
{
  "schema_version": 1,
  "id": "IR-2026-001",
  "analysis_date": "2026-06-17",
  "product": "Продукт А",
  "batch": "П-2026-04",
  "sample_name": "Образец фильтрата",
  "description": "ИК-анализ фильтрата после стадии очистки",
  "short_result": "Соответствует эталону",
  "status": "готов",
  "comment": "",
  "attachments": {
    "photos":  ["photos/photo_1.jpg", "photos/photo_2.jpg"],
    "spectra": ["spectra/spectrum_1.pdf", "spectra/spectrum_1.txt"],
    "reports": ["reports/report.pdf"]
  },
  "created_at": "2026-06-17T10:15:00",
  "updated_at": "2026-06-17T10:42:00",
  "committed": true
}
```

Пояснения:

- `schema_version` — версия формата карточки (для будущих миграций).
- Пути во вложениях — **относительные к папке анализа** (раздел 6).
- `committed` — флаг завершённости операции создания (см. ARCHITECTURE.md, §5–6).
- `created_at` / `updated_at` соответствуют колонкам «дата создания»/«дата изменения».

## 5. Правила именования ID анализа

Формат: **`IR-YYYY-NNN`**

- `IR` — префикс типа учёта (InfraRed / ИК-спектры). Постоянный для текущей
  предметной области; в будущем может стать настраиваемым.
- `YYYY` — год создания анализа (4 цифры), например `2026`.
- `NNN` — порядковый номер **внутри года**, начиная с `001`, не менее 3 цифр.
  При переполнении 999 номер расширяется до 4 цифр (`1000`).

Правила генерации:

1. Номер выдаётся приложением; пользователь его не вводит.
2. Уникальность проверяется по **двум источникам сразу**: существующим папкам
   `samples/IR-YYYY-*` и строкам `registry.xlsx`. Берётся максимум + 1.
3. Счётчик **сбрасывается каждый новый год** (`IR-2027-001`).
4. ID **неизменяем** после создания — он же имя папки и ключ строки Excel.
5. ID — валидное имя папки во всех ОС (только латиница, цифры, дефис).

## 6. Правила относительных путей

1. Все пути в `card.json` и в `registry.xlsx` хранятся **относительно**:
   - вложения в `card.json` — относительно **папки анализа**
     (`photos/photo_1.jpg`);
   - «путь к папке анализа» в Excel — относительно **`lab_data/`**
     (`samples/IR-2026-001`);
   - ссылки на файлы в Excel — относительно `lab_data/`
     (`samples/IR-2026-001/photos/photo_1.jpg`).
2. **Абсолютные пути не хранятся** — чтобы базу можно было перенести на другой
   диск/компьютер без правок.
3. Разделитель путей в файлах — прямой слэш `/` (кросс-платформенно).
   Приложение конвертирует в системный разделитель при открытии файлов.
4. Несколько ссылок в одной ячейке Excel разделяются `;`.
5. При открытии файла приложение строит абсолютный путь как
   `корень lab_data` + относительный путь.

## 7. Правила хранения фотографий, спектров и отчётов

Общее:

- Файлы **копируются** в папку анализа (оригиналы пользователя не трогаются).
- Имена нормализуются приложением; пробелы и нелатинские символы в служебном
  имени не используются. Исходное имя при необходимости можно сохранить в
  `card.json` как метаданные.
- Расширение сохраняется от исходного файла.

**Фотографии — `photos/`:**
- Имя: `photo_N.<ext>` (`photo_1.jpg`, `photo_2.png`), где `N` — порядковый
  номер в пределах анализа.
- Допустимые форматы (MVP): `jpg`, `jpeg`, `png`.

**Спектры — `spectra/`:**
- Имя: `spectrum_N.<ext>` (`spectrum_1.pdf`, `spectrum_1.txt`).
- Один логический спектр может иметь несколько файлов с **одинаковым номером**
  и разными расширениями (`spectrum_1.pdf` + `spectrum_1.txt`) — это PDF-вид
  и текстовые данные одного измерения.
- Допустимые форматы (MVP): `pdf`, `txt`.

**Отчёты — `reports/`:**
- Имя: `report.pdf` для основного отчёта; при нескольких — `report_N.pdf`.
- Допустимые форматы (MVP): `pdf` (при необходимости `docx`).

Согласованность:

- После любого добавления/удаления файла приложение обновляет списки
  вложений **и в `card.json`, и в `registry.xlsx`** (сначала card.json).
- Удаление файла — **мягкое**: файл переносится в `.trash/` с сохранением
  структуры, ссылки убираются из карточки и реестра.
