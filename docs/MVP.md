# LabSpectra — MVP (минимально жизнеспособная версия)

MVP строится вокруг трёх вещей: **папки анализов**, **`card.json`** и
**`registry.xlsx`**. Приложение — единственная точка управления данными.
Сложная база данных в MVP **не нужна**; она возможна позже как опциональный
индекс, если Excel и папок станет недостаточно.

---

## 1. Что обязательно должно быть в первой версии

**Хранилище и данные**
- Корень `lab_data/` со структурой `samples/`, `backups/`, служебными папками.
- Создание папки анализа с подпапками `photos/`, `spectra/`, `reports/`.
- `card.json` как источник истины по каждому анализу.
- `registry.xlsx` как человекочитаемый реестр (15 колонок из STORAGE_MODEL §3).
- Генерация ID анализа `IR-YYYY-NNN` приложением (без ручного ввода).
- Относительные пути ко всем файлам (переносимость базы).

**Операции (через приложение)**
- Создать анализ (форма → папка + card.json + строка Excel).
- Добавить фотографии, спектры, отчёты (копирование + переименование).
- Редактировать поля анализа (продукт, партия, описание, результат, статус,
  комментарий) с обновлением card.json и Excel.
- Открыть файл штатной программой ОС и открыть папку анализа в Проводнике.

**Целостность**
- Запись в фиксированном порядке: сначала `card.json`, затем `registry.xlsx`.
- Проверка согласованности при старте (reconcile) и команда
  **«Пересобрать registry.xlsx из card.json»**.
- Хотя бы базовая защита: снимок `registry.xlsx` перед перезаписью.

**Поиск (минимум)**
- Поиск по ID/продукту/партии и фильтр по статусу.

**UI (минимум)**
- Список анализов, форма создания, карточка анализа с вложениями,
  понятные сообщения об ошибках.

**Сборка**
- Запускаемый локально билд под Windows; выбор папки `lab_data` при первом старте.

## 2. Что можно отложить

- **Полноценная СУБД / индекс (SQLite и т.п.)** — только если объёмы перерастут
  возможности Excel; в MVP не входит.
- Расширенный поиск: полнотекстовый, по содержимому отчётов, сложные комбинации фильтров.
- Просмотр/построение графиков ИК-спектров из сырых данных (`.txt`).
- Встроенный просмотрщик фото/PDF (в MVP — открытие внешней программой).
- Автоматические бэкапы по расписанию и тонкая политика ротации
  (в MVP достаточно ручного бэкапа + снимок перед записью).
- Импорт уже существующих папок/таблиц, миграции из старых форматов.
- Многопользовательский режим, сетевой/общий доступ, права ролей.
- Кросс-платформенные билды (Linux/macOS) — начать с Windows.
- Теги, произвольные пользовательские поля, шаблоны анализов.
- Тёмная тема, локализация интерфейса на другие языки.

## 3. Риски подхода «Excel + папки»

1. **Рассинхронизация Excel и папок.**
   Строка в реестре есть, а папки нет (или наоборот); ссылки указывают на
   отсутствующие файлы.

2. **Повреждение `registry.xlsx`.**
   Файл может побиться при сбое записи/диска или стать нечитаемым.

3. **Файл занят другой программой.**
   Пользователь открыл `registry.xlsx` в Excel — приложение не может записать.

4. **Ручные правки в обход приложения.**
   Пользователь меняет строки/пути в Excel или переименовывает папки в
   Проводнике, нарушая инварианты.

5. **Конкурентный доступ.**
   Два экземпляра приложения или пользователь + приложение пишут одновременно.

6. **Масштаб и производительность.**
   При тысячах строк Excel и список в памяти могут стать медленными.

7. **Переносимость путей.**
   Абсолютные пути сломались бы при переносе базы на другой диск/ПК.

8. **Потеря данных.**
   Случайное удаление файлов/папок, отсутствие резервных копий.

---

## 4. Как уменьшить эти риски

| Риск | Меры снижения |
|------|---------------|
| 1. Рассинхронизация | `card.json` — источник истины; reconcile при старте; команда пересборки реестра; единая точка записи (сервис). |
| 2. Повреждение Excel | Реестр **производный** — пересобирается из всех `card.json`; битый файл сохраняется как `registry.corrupted-…`; восстановление из `backups/`. |
| 3. Файл занят | Обнаружение блокировки, понятное сообщение «закройте registry.xlsx»; запись после освобождения; данные при этом уже в `card.json`. |
| 4. Ручные правки | Позиционировать Excel как **просмотр**, а не способ ввода; reconcile подсветит расхождения и предложит починку из card.json; ID/папки неизменяемы. |
| 5. Конкурентный доступ | Lock-файл `lab_data/.lock`; предупреждение о втором экземпляре; запись операций целиком. |
| 6. Масштаб | В MVP — список в памяти (достаточно для лаборатории); при росте — опциональный индекс (SQLite) **поверх** тех же папок, без смены источника истины. |
| 7. Пути | Хранить только **относительные** пути; разделитель `/`; собирать абсолютный путь из корня при открытии. |
| 8. Потеря данных | Снимок `registry.xlsx` перед перезаписью; ручной полный бэкап `lab_data/` в `backups/`; мягкое удаление в `.trash/`. |

**Главный страховочный механизм:** реестр Excel всегда восстановим из
совокупности `card.json`. Поэтому даже при потере или повреждении
`registry.xlsx` данные анализов не теряются — индекс перестраивается.
