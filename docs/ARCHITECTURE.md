# LabSpectra — Архитектура

## 1. Цель системы

**LabSpectra** — локальное настольное приложение для химической лаборатории.
Оно ведёт учёт анализов: ИК-спектров, фотографий образцов, отчётов и текстовых
результатов.

Ключевые цели:

- Хранить данные анализов в **обычных папках**, доступных через Проводник.
- Вести **человекочитаемый реестр** `registry.xlsx`, который можно открыть без приложения.
- Сделать приложение **единственной точкой управления данными**: пользователь не
  создаёт папки и строки Excel руками — это делает программа.
- Гарантировать, что папки на диске и строки в Excel **не расходятся** между собой.
- Работать полностью **локально**, без сервера и без обязательной СУБД.

Чего система сознательно **не делает** в первой версии:

- не использует «тяжёлую» базу данных как обязательный компонент;
- не синхронизируется с облаком;
- не обрабатывает спектры (не строит графики из сырых данных) — только хранит файлы.

## 2. Основные пользовательские сценарии

1. **Создание анализа.**
   Пользователь нажимает «Новый анализ», заполняет форму (продукт, партия,
   название образца, описание, результат, статус, комментарий). Приложение:
   - генерирует `ID анализа` (например, `IR-2026-001`);
   - создаёт папку `lab_data/samples/IR-2026-001/` с подпапками `photos/`,
     `spectra/`, `reports/`;
   - сохраняет `card.json`;
   - добавляет строку в `registry.xlsx`.

2. **Добавление файлов к анализу.**
   Пользователь выбирает фотографии, спектры (PDF/TXT) и отчёты. Приложение
   копирует их в соответствующие подпапки анализа, переименовывает по правилам
   (`photo_1.jpg`, `spectrum_1.pdf`) и обновляет ссылки в `card.json` и в Excel.

3. **Просмотр и редактирование анализа.**
   Пользователь открывает карточку анализа, меняет описание/результат/статус.
   Приложение обновляет `card.json`, строку в `registry.xlsx` и поле
   «дата изменения».

4. **Поиск и фильтрация.**
   Пользователь ищет анализы по ID, продукту, партии, статусу, дате.

5. **Открытие файлов.**
   Из карточки пользователь открывает файл штатной программой ОС или
   переходит в папку анализа в Проводнике.

6. **Резервное копирование.**
   Пользователь (или авто-режим) создаёт резервную копию `lab_data` в `backups/`.

## 3. Слои приложения

Приложение разбивается на чётко разделённые слои. Зависимости направлены
сверху вниз: верхние слои знают о нижних, нижние о верхних — нет.

```
┌─────────────────────────────────────────────┐
│  UI layer (presentation)                     │  Fyne / Wails
│  формы, списки, диалоги, обработка действий   │
├─────────────────────────────────────────────┤
│  Application / Service layer (бизнес-логика)  │  сценарии: создать,
│  оркестрация операций, транзакционность       │  добавить файл, изменить
├─────────────────────────────────────────────┤
│  Domain layer (доменная модель)              │  Analysis, Sample,
│  сущности, правила, генерация ID             │  Attachment, Status
├───────────────┬───────────────┬─────────────┤
│ FileStorage   │ ExcelRegistry │ CardStore    │  инфраструктура
│ (папки/файлы) │ (registry.xlsx)│ (card.json) │
├───────────────┴───────────────┴─────────────┤
│  BackupService            │  Config / Paths   │
└─────────────────────────────────────────────┘
```

### 3.1 UI layer (презентация)
- Отвечает **только** за отображение и ввод.
- Не пишет файлы и не трогает Excel напрямую.
- Вызывает методы сервисного слоя и показывает результат/ошибки.
- Технологии-кандидаты для Go: **Fyne** (чистый Go, нативный UI) или
  **Wails** (Go + веб-фронтенд). Выбор фиксируется на этапе UI (см. план).

### 3.2 Application / Service layer (бизнес-логика)
- Реализует пользовательские сценарии как **единые операции**:
  `CreateAnalysis`, `AddPhoto`, `AddSpectrum`, `AddReport`, `UpdateAnalysis`,
  `Search`, `Backup`.
- **Оркеструет** запись: в каком порядке трогать файлы, card.json и Excel.
- Обеспечивает атомарность на уровне операции (см. раздел 5–6).
- Единственный слой, которому разрешено менять данные. UI ходит только сюда.

### 3.3 Domain layer (доменная модель)
- Сущности: `Analysis` (анализ/образец), `Attachment` (фото/спектр/отчёт),
  `AnalysisStatus`, `AnalysisID`.
- Правила: генерация ID, валидация полей, нормализация имён файлов.
- Не зависит от Excel, файловой системы и UI — это «чистый» код.

### 3.4 Инфраструктура
- **FileStorage** — создаёт/удаляет папки анализов, копирует и переименовывает
  фотографии, спектры, отчёты, строит относительные пути.
- **ExcelRegistry** — читает и пишет `registry.xlsx` (кандидат-библиотека для
  Go: `github.com/xuri/excelize/v2`). Знает порядок и состав колонок,
  умеет находить строку по ID и обновлять её.
- **CardStore** — сериализует/десериализует `card.json` (стандартный
  `encoding/json`).
- **BackupService** — упаковывает `lab_data` в архив/копию в `backups/`.
- **Config / Paths** — хранит корневой путь к `lab_data`, версию схемы,
  настройки приложения.

## 4. Кто за что отвечает (ответственность по данным)

| Данные                     | Владелец (слой)         | Источник истины |
|----------------------------|-------------------------|-----------------|
| Поля анализа               | Service → CardStore     | `card.json`     |
| Человекочитаемый реестр    | Service → ExcelRegistry | `registry.xlsx` |
| Сами файлы (фото/спектры)  | Service → FileStorage   | папки анализа   |
| Резервные копии            | BackupService           | `backups/`      |
| Отображение                | UI                      | —               |

**Главный принцип источника истины:**
для восстановления данных **истиной считаются папки + `card.json`**.
`registry.xlsx` — это **производный человекочитаемый индекс**, который всегда
можно перестроить из совокупности `card.json`. Это решает проблему повреждения
Excel (раздел 7) и рассинхронизации (раздел 6).

## 5. Порядок записи данных

Любая операция изменения данных выполняется сервисным слоем в строгом порядке,
от «самого надёжного» к «производному»:

**Создание анализа:**
1. Сгенерировать `ID` (с проверкой уникальности по существующим папкам и Excel).
2. Создать папку анализа и подпапки во **временном** виде или сразу с пометкой
   «черновик» в `card.json` (`"status": "draft"` / флаг `committed: false`).
3. Записать `card.json` (полная карточка) — **источник истины фиксируется первым**.
4. Добавить строку в `registry.xlsx`.
5. Снять флаг черновика (`committed: true`) в `card.json`.

**Добавление файла:**
1. Скопировать файл во временное имя внутри нужной подпапки.
2. Переименовать в финальное имя (`photo_2.jpg`) — атомарная операция ОС.
3. Обновить список вложений в `card.json`.
4. Обновить ссылки в строке `registry.xlsx`.

**Изменение полей:**
1. Обновить `card.json` (+ «дата изменения»).
2. Обновить строку в `registry.xlsx`.

Правило: **сначала файлы и `card.json` (истина), затем `registry.xlsx` (индекс)**.
Если шаг с Excel упал — данные не потеряны, индекс чинится пересборкой.

## 6. Как избежать рассинхронизации Excel и папок

1. **Единая точка записи.** Менять данные может только сервисный слой.
   Прямой ручной ввод строк в Excel — не основной сценарий (см. MVP, риски).

2. **Excel — производный индекс, а не истина.** Любая строка `registry.xlsx`
   может быть восстановлена из соответствующего `card.json`.

3. **Операция как единое целое.** Сервис выполняет шаги последовательно;
   при ошибке на шаге Excel операция помечается незавершённой, а не
   «полузаписанной молча».

4. **Флаг `committed` в `card.json`.** Незакоммиченные карточки (приложение
   упало между шагами) обнаруживаются при старте и до-записываются в Excel
   или откатываются.

5. **Проверка целостности при запуске («reconcile»).** Приложение сверяет:
   - для каждой папки `samples/*` есть ли строка в Excel;
   - для каждой строки Excel существует ли папка;
   - совпадают ли ключевые поля и ссылки на файлы.
   Расхождения показываются пользователю с предложением «починить из card.json».

6. **Блокировка от параллельного доступа.** На время записи `registry.xlsx`
   приложение держит lock-файл (`lab_data/.lock`) и предупреждает, если Excel
   открыт в другой программе (файл заблокирован на запись).

7. **Стабильный формат Excel.** Фиксированный порядок и имена колонок,
   версия схемы хранится в `card.json` и в служебной ячейке/листе Excel.

## 7. Что делать, если Excel повреждён

`registry.xlsx` — восстановимый индекс, поэтому повреждение не критично.

Стратегия:

1. **Обнаружение.** При старте приложение пытается открыть `registry.xlsx`.
   Если файл не читается / не той схемы — переход в режим восстановления.

2. **Сохранение повреждённого файла.** Битый файл переименовывается в
   `registry.corrupted-YYYYMMDD-HHMMSS.xlsx` и не удаляется (для разбора).

3. **Восстановление из резервной копии.** Если в `backups/` есть свежая
   валидная копия `registry.xlsx` — предложить восстановить её.

4. **Полная пересборка из `card.json`.** Приложение обходит все
   `samples/*/card.json`, читает их и **заново генерирует** `registry.xlsx`
   со всеми строками, ссылками и колонками. Это главный механизм:
   реестр всегда выводим из карточек.

5. **Отчёт пользователю.** Показать, сколько анализов восстановлено, какие
   карточки оказались битыми (требуют ручного внимания).

Тот же механизм пересборки используется как сервисная команда
«Пересобрать реестр» даже без повреждения — для гарантии согласованности.

## 8. Обработка ошибок и логирование

- Каждая операция сервисного слоя возвращает явную ошибку; UI показывает
  понятное сообщение, не молчит.
- Действия, меняющие данные, пишутся в лог (`lab_data/logs/app.log`):
  что, когда, по какому ID.
- Перед перезаписью `registry.xlsx` создаётся быстрый снимок предыдущей
  версии (в `backups/` или `lab_data/.trash/`), чтобы откатиться при сбое.
