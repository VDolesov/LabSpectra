# LabSpectra — План разработки

План разбит на этапы. Каждый этап даёт проверяемый результат и закрывается
тестами, прежде чем начинается следующий. Стек — **Go** (модуль `go 1.26.x`).

Принцип последовательности: сначала надёжное ядро хранения (папки + `card.json`),
затем производный реестр Excel, затем поиск, бэкапы и только потом UI. Так UI
строится поверх уже работающей и проверенной логики.

---

## Этап 1. Проектирование доменной модели

**Цель:** зафиксировать сущности и правила, не зависящие от Excel/файлов/UI.

Задачи:
- Описать сущности: `Analysis`, `Attachment` (с типом photo/spectrum/report),
  `AnalysisStatus`, `AnalysisID`.
- Реализовать генерацию и валидацию `AnalysisID` (`IR-YYYY-NNN`, см. STORAGE_MODEL §5).
- Правила валидации полей (обязательные/необязательные, форматы дат).
- Нормализация имён файлов и относительных путей.
- Версия схемы (`schema_version`).

Результат: пакет `domain/` без внешних зависимостей.
Проверка: unit-тесты на генерацию ID, валидацию, нормализацию путей.

---

## Этап 2. Реализация файлового хранилища

**Цель:** управлять папками и файлами анализов.

Задачи:
- Создание корня `lab_data/` и служебных папок (`samples/`, `backups/`,
  `logs/`, `.trash/`).
- Создание папки анализа с подпапками `photos/`, `spectra/`, `reports/`.
- Копирование + переименование вложений (`photo_N`, `spectrum_N`, `report`).
- Построение и разбор относительных путей; открытие файла штатной программой ОС.
- Мягкое удаление в `.trash/`.
- Lock-файл `.lock` на время операций записи.

Результат: пакет `storage/file/`.
Проверка: тесты на временной директории (создание анализа, добавление файлов,
переименование, мягкое удаление, перенос корня без потери ссылок).

---

## Этап 3. Реализация работы с Excel

**Цель:** вести `registry.xlsx` как производный человекочитаемый индекс.

Задачи:
- Подключить библиотеку Excel для Go (кандидат: `github.com/xuri/excelize/v2`).
- Создание `registry.xlsx` со строкой заголовков (15 колонок, см. STORAGE_MODEL §3)
  и служебной информацией о версии схемы.
- Добавление строки анализа, обновление строки по ID, чтение всех строк.
- Запись ссылок на файлы (относительные пути, разделитель `;`).
- Безопасная перезапись: снимок предыдущей версии перед записью.
- Обработка «файл занят другой программой».

Результат: пакет `storage/excel/`.
Проверка: тесты на запись/чтение/обновление строк, проверка порядка колонок.

---

## Этап 4. Реализация карточки анализа (`card.json`) и сервисного слоя

**Цель:** связать домен, файлы и Excel в целостные операции.

Задачи:
- `CardStore`: сериализация/десериализация `card.json` (`encoding/json`),
  поле `committed`, `schema_version`.
- Сервисные операции (оркестрация в правильном порядке, см. ARCHITECTURE §5):
  `CreateAnalysis`, `AddPhoto`, `AddSpectrum`, `AddReport`, `UpdateAnalysis`,
  `DeleteAttachment`.
- Согласованная запись: сначала `card.json` (истина), затем `registry.xlsx`.
- **Reconcile**: сверка папок и Excel при старте; восстановление недописанных
  (`committed=false`) операций.
- **Пересборка `registry.xlsx` из всех `card.json`** (восстановление при
  повреждении Excel, см. ARCHITECTURE §7) — отдельная сервисная команда.

Результат: пакет `service/` поверх `domain/`, `storage/file/`, `storage/excel/`.
Проверка: интеграционные тесты полного цикла + сценарий «сломали Excel —
пересобрали из card.json — данные совпали».

---

## Этап 5. Реализация поиска и фильтрации

**Цель:** быстро находить анализы.

Задачи:
- Загрузка списка анализов (из `registry.xlsx`, при необходимости сверка с card.json).
- Поиск по ID, продукту, партии, названию образца.
- Фильтры по статусу и диапазону дат.
- Сортировка (по дате, по ID).
- Для MVP — поиск в памяти по списку; индекс/СУБД не требуются.

Результат: пакет `service/query/` (или метод сервиса).
Проверка: тесты на выборки и фильтры.

---

## Этап 6. Реализация резервного копирования

**Цель:** защитить данные от потери и повреждения Excel.

Задачи:
- Копия `registry.xlsx` перед каждой перезаписью (быстрый снимок).
- Полный бэкап `lab_data/` в `backups/` (архив с датой в имени).
- Политика хранения (например, последние N копий) и ручной запуск из UI.
- Восстановление из выбранной резервной копии.

Результат: пакет `service/backup/`.
Проверка: тесты на создание/ротацию/восстановление копий.

---

## Этап 7. Реализация UI

**Цель:** дать пользователю единый интерфейс управления данными.

Задачи:
- Выбрать фреймворк: **Fyne** (чистый Go) или **Wails** (Go + веб-фронтенд).
- Экран списка анализов с поиском/фильтрами.
- Форма создания анализа.
- Карточка анализа: просмотр/редактирование полей, добавление и открытие
  фотографий/спектров/отчётов, переход в папку в Проводнике.
- Диалоги резервного копирования и восстановления.
- Индикация ошибок и результата операций; экран восстановления при битом Excel.
- UI вызывает **только** сервисный слой, напрямую файлы/Excel не трогает.

Результат: пакет `ui/`.
Проверка: ручное прохождение всех пользовательских сценариев из ARCHITECTURE §2.

---

## Этап 8. Сборка локального приложения

**Цель:** получить готовый к запуску локальный продукт.

Задачи:
- Сборка под целевую ОС (Windows) — единый исполняемый файл.
- Первичная инициализация: выбор/создание папки `lab_data` при первом запуске.
- Конфигурация (путь к данным, политика бэкапов) в файле настроек.
- Логирование в `lab_data/logs/app.log`.
- Краткая инструкция пользователя (README): как запустить, где лежат данные,
  как сделать бэкап.
- Проверка переносимости: скопировать `lab_data` на другую машину — данные
  открываются без правок (относительные пути).

Результат: распространяемый билд + инструкция.
Проверка: чистая установка на «пустой» машине, полный цикл работы.

---

## Сквозные практики (на всех этапах)

- Тесты пишутся вместе с кодом этапа; следующий этап не начинается без зелёных тестов.
- `card.json` + папки — источник истины; `registry.xlsx` — производный индекс.
- Любая запись данных идёт через сервисный слой в фиксированном порядке.
- Версионирование схемы (`schema_version`) с самого начала — для будущих миграций.
